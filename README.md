#### Запуск 
```docker-compose up -d```

#### Документация
<http://localhost:8000/docs> после запуска

#### Примечания
Проект тестовый, поэтому есть некоторые допущения.
1. Код в сервисах может быть не оптимизирован как по стилю так и по производительности
2. Не выведены переменные окружения наружу (настройки сервисов находятся в _src/services/<service_name>/src/config.py_)
3. В документации API могут отсутствовать схемы выходных данных
4. Нет документации в отдельных сервисах. только общая
5. Не все критические ошибки обработаны (При сценарии когда отваливается что то из инфраструктуры проект не сможет сам себя восстановить или выдать информативное уведомление о поломке, только ошибка 500)
6. RESTApi интерфейсы внутренних сервисов с OpenAPI выставлены наружу (по дефолту на портах 8001, 8002, 8003, 8004. В проде соответственно следует убрать)
7. У сервисов нет системы обнаружения (хотя сначала так задумывалось). Их адреса жестко прописаны в настройках gateway
8. Некоторые выводы конечных точек могут содержать только ID сущностей (например только ID продукта если посмотреть корзину или список заказов)
Лечится простым дублированием инфы в базе микросервиса с асинхронным обновлением при необходимости (тут такой задачи не было)

#### Особенности
1. Основной сервис - __gateway__. Он принимает все запросы и маршрутизирует их в нужный сервис.
Отвечает за аутентификацию пользователей
2. Остальные сервисы внутренние. Каждый отвечает за свою область, хранит только свои данные
3. Асинхронное взаимодействие (через RabbitMQ) может идти в обход __gateway__.
Использован подход слепой трансляции через TOPIC с _routing_key_ с подписанными слушателями
4. Цены на товары хранятся как тип _biginteger_ (цена в копейках или в любом другом порядке, конкретную точность можно указать на фронте.)
Это хороший удобный способ хранить вещественные числа с заранее известной точностью (в данном случае 2 знака после точки)
